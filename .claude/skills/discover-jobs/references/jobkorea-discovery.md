# 잡코리아 공고 발견(검색→매칭→스크랩) — 상세 절차

`discover-jobs` 스킬에서 잡코리아(jobkorea.co.kr)를 능동 탐색할 때 사용한다. 마스터 이력서 검색어로 공고를 검색하고, 매칭되는(높음·중간) 공고를 **스크랩에 추가**한다. 검증된 방식은 2026-06-23 크롬 정찰에서 확립되었다.

저장한 스크랩의 회사 목록 수집(재무 스크리닝용)은 `screen-companies/references/jobkorea-scraps.md`(경로 B)에 있다 — 발견은 그 앞단이다.

## 핵심 원리

- 잡코리아 검색은 **모던 SPA(Tailwind 클래스)** 라, 카드 클래스가 난독화돼 셀렉터가 불안정하다. **공고번호는 `a[href*="GI_Read/{공고번호}"]` 링크에서 추출**하는 게 가장 안정적이다.
- 스크랩 추가/해제는 레거시 GET 엔드포인트로 처리된다. 다만 부가 파라미터(`Gno`·`Mem_Type_Code`) 도출이 까다로워, **카드의 스크랩 버튼을 프로그램적으로 클릭**하는 방식을 1순위로 권장한다(페이지 JS가 올바른 요청을 만든다).
- **로그인 필수**. 로그인 시 홈/검색에 "○○님" 개인화가 보인다.
- 회사명이 목록에서 잘려 보일 수 있다 → 경로 B처럼 `GI_Read/{공고번호}` 상세의 `og:title`로 복원한다.
- ⚠️ 잡코리아는 스크랩 시 별도 추적 POST(`/Search/api/dmp/scrap`)도 보내는데, 그 바디에 **회원번호·성별·나이 등 개인정보가 포함**된다. 이 추적 호출은 발견에 불필요하니 **건드리지 말고, 그 데이터를 로그·파일에 절대 저장하지 말 것**.

## 1. 크롬 연결·로그인 확인 (전제)

1. 브라우저 도구 로드 후 `tabs_context_mcp` 확인. 새 탭에서 `https://www.jobkorea.co.kr/` 를 열고 "○○님" 개인화가 보이면 로그인됨. 아니면 잡코리아 로그인 요청 후 건너뜀.

## 2. 검색 실행

### 검색 URL
```
https://www.jobkorea.co.kr/Search/?stext={검색어}&tabType=recruit
```
- `stext`: 마스터 이력서 검색어(백엔드, AI, AI Agent, 풀스택, Python, FastAPI, Django 등).
- 직무·경력·지역 필터 UI(직무/경력/기업형태/학력/고용형태 드롭다운)가 있으나, 경력은 카드/상세에서 파싱해 클라이언트 필터하는 게 단순하다.
- 검색 상단의 "나에게 꼭 맞는 공고/지금 주목할 만한 공고"는 **AD(광고)** 이므로 매칭 대상에서 제외(카드에 `AD` 라벨).

### 결과 파싱
`javascript_tool`로:
- 공고번호: `[...document.querySelectorAll('a[href*="GI_Read"]')]` 에서 `href`의 `GI_Read/(\d+)` 캡처 → 중복 제거(Set). (페이지당 ~20여 건.)
- 회사명·공고명·마감(D-day)·경력: 각 GI_Read 링크의 카드 컨테이너 `innerText`를 줄 단위로 파싱하거나, 불안정하면 공고번호로 상세 `GI_Read/{no}`의 `og:title`(회사·공고명)·본문(경력·마감)을 읽어 보강.
- ⚠️ 출력에 URL/쿼리스트링 포함 시 도구가 [BLOCKED] 처리 → **텍스트만** 반환(공고번호는 `#` 치환 등).

## 3. 매칭·하드 필터

- 경력: 카드/상세의 경력 하한이 3년 초과면 제외(주니어~3년). "신입"·"경력무관" 포함.
- 직무: 공고명·직무 태그로 백엔드·AI·Agent·풀스택만. 제외: 프론트 전용·데이터엔지니어·QA·기획 등.
- 마감: D-day가 지난 공고 제외("상시"·"채용시"는 포함).
- 등급은 공고명·직무를 마스터 이력서와 대조(기술스택 미노출 시 제목·분야 기반).

## 4. 자동 스크랩

**1순위 — 카드 스크랩 버튼 클릭(권장):** 각 대상 공고 카드의 스크랩 아이콘(`i.icon--system_scrap`)의 클릭 가능한 조상 요소를 찾아 `.click()` 디스패치. 페이지 JS가 올바른 GI_Scrap 요청을 만들어 준다. 클릭 후 별이 채워졌는지/스크랩 목록 반영으로 검증.

**기반 메커니즘(참고) — GI_Scrap GET:**
```
GET https://www.jobkorea.co.kr/Recruit/GI_Scrap?GI_No={공고번호}&Gno={Gno}&Mem_Type_Code={코드}&Scrap_State=1   // 추가
                                                                                          &Scrap_State=0   // 해제
```
- `Scrap_State` = **1(추가) / 0(해제)** 로 확정. `Mem_Type_Code`(예: `C`)·`Gno`는 카드/페이지에서 와야 하므로, 직접 URL 조립보다 위 클릭 방식이 안전하다.
- 실패 시 해당 공고 "스크랩 실패"로 표기하고 계속.

## 5. 중복 제거 — 이미 스크랩/거른 회사

- **이미 스크랩:** 경로 B(`jobkorea-scraps.md`)의 스크랩 목록 `https://www.jobkorea.co.kr/User/Scrap?PageSize=100` 에서 기존 공고번호 집합을 모아 대조해 제외.
- **이미 거른 회사:** 회사명 정규화(`㈜`·`(주)`·`주식회사`·영문 병기 괄호 제거) 후 `data/company-screening/*.md` ❌거름 목록과 대조해 제외. (잘린 회사명은 og:title로 복원 후 비교.)

## 에러 핸들링

- 로그인 안 됨 → 잡코리아 건너뛰고 사용자에게 로그인 요청.
- 카드 파싱 실패가 잦으면 공고번호만 모아 상세 `GI_Read/{no}`로 보강.
- 특정 공고 스크랩 실패 → 그 공고만 표기 후 계속.
- alert/confirm 유발 클릭 금지. 2~3회 연속 실패 시 멈추고 보고.

## 정찰 메모 (2026-06-23 확립)

- 스크랩 실제 동작은 `GET /Recruit/GI_Scrap`(파라미터 `GI_No`·`Gno`·`Mem_Type_Code`·`Scrap_State`), 별도 `POST /Search/api/dmp/scrap`(추적, `scrapType:ADD`/개인정보 포함 — 무시).
- `Scrap_State` 토글 확인: 추가 1 / 해제 0. 테스트 스크랩은 0으로 즉시 원상복구.
- 모던 SPA라 카드 셀렉터가 불안정 → 공고번호는 GI_Read 링크에서, 스크랩은 버튼 클릭 디스패치로 처리하는 게 견고하다. **4사 통합 검증(Task 7) 때 실제 검색→스크랩 흐름을 라이브로 한 번 확인할 것.**
