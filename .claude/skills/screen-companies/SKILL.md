# screen-companies — 회사 지원 여부 스크리닝

사용자가 붙여넣은 회사 도메인/회사명을 캐치(catch.co.kr)에서 조회해, 재무·인원 기준으로 **지원할 가치가 있는지** 판정한다.

## 거름(❌) 기준

아래 중 **하나라도** 해당하면 거름:

1. **적자 + 매출 정체/감소**: 영업이익이 적자(마이너스)인데 매출이 지속적으로 증가하지 못함.
   - 단, 적자여도 **매출이 지속 증가**하면 → 지원 가능(성장 중인 적자는 정상).
   - "지속 증가"는 최근 확보 가능한 매출 데이터(보통 3개년)가 우상향이면 충족.
2. **인원 5명 이하**: 직원수가 5명 이하.

위 어디에도 안 걸리면 **✅지원**.
판정에 필요한 데이터를 못 찾으면 **⚠️확인불가**로 표기(임의 추정 금지).

## 입력 경로 (둘 중 하나)

스크리닝 대상 회사 목록을 어떻게 확보하는지에 따라 두 경로가 있다. 사용자 요청에서 판단한다.

- **경로 A — 직접 붙여넣기**: 사용자가 채팅에 회사 도메인/회사명을 줄줄이 붙여넣음. → 아래 "실행 절차(경로 A)"를 따른다. 데이터 출처는 캐치(catch.co.kr).
- **경로 B — 잡사이트 저장 공고 자동 수집**: 사용자가 "(사이트) 북마크/스크랩/저장한 공고 정리", "크롬에서 저장한 공고 찾아 스크리닝", "내 저장 공고 다 걸러줘", "관심 기업 걸러줘" 등을 요청. **크롬 연결이 전제**다(사용자가 해당 사이트에 로그인된 세션). → 저장 공고 목록을 자동 수집한 뒤 스크리닝한다.
  - 크롬이 연결돼 있지 않거나 해당 사이트 로그인 세션이 아니면 자동 수집이 불가하므로, 사용자에게 알리고 경로 A(회사명 직접 입력)로 폴백한다.

### 경로 B 사이트 라우팅

요청에서 사이트를 판단해 해당 reference 문서를 읽고 그 절차를 따른다. 사이트를 특정하지 않고 "내 저장 공고 다"·"전부"라고 하면 **로그인된 사이트 전부**를 순회한다(각 사이트 절차를 차례로 실행 후 통합 출력).

| 사이트 | 트리거 예 | 저장 공고 수집 | 재무·인원 출처 | 상세 문서 |
|--------|-----------|----------------|----------------|-----------|
| 원티드 | "원티드 북마크" | `/profile/bookmarks` | **원티드 자체 데이터** | `references/wanted-bookmarks.md` |
| 사람인 | "사람인 스크랩" | `scrap-recruit` 페이지 | 캐치(일원화) | `references/saramin-scraps.md` |
| 잡코리아 | "잡코리아 스크랩" | `/User/Scrap` 페이지 | 캐치(일원화) | `references/jobkorea-scraps.md` |
| 점핏 | "점핏 스크랩/북마크" | `scrap/list` API | 캐치(일원화) | `references/jumpit-bookmarks.md` |

> **재무 데이터 일원화:** 원티드만 자체 재무 API를 쓴다(이미 확립). **사람인·잡코리아·점핏은 회사명·마감일만 수집**하고, 재무·인원은 수집한 회사명을 아래 **"실행 절차(경로 A) 2단계(캐치 조회)"** 로 넘겨 동일하게 판정한다(캐치는 사람인 그룹의 기업정보라 안정적). 회사명은 캐치 검색 전 `㈜`·`(주)`·`주식회사`·영문 병기 괄호를 제거해 정규화한다.

판정 기준("거름 기준")은 모든 경로가 **동일**하다. 경로 B는 회사 목록 수집·마감일 수집과(원티드 한정으로) 데이터 출처만 다르다.

### 경로 A — 직접 붙여넣은 회사 목록 처리

- 붙여넣은 항목을 한 줄(또는 공백/쉼표) 단위로 분리해 리스트화한다.
- 도메인(`example.com`)이면 → 회사명 추정이 필요. 캐치 검색창에 도메인 또는 회사명으로 검색.
- 도메인에서 회사명이 불명확하면, 먼저 해당 도메인을 방문해 회사명을 파악한 뒤 캐치에서 검색.

## 실행 절차 (경로 A)

### 1. 브라우저 준비
1. `ToolSearch`로 `mcp__claude-in-chrome__tabs_context_mcp` 로드 → 호출해 현재 탭 상황 파악.
2. 새 탭 생성(`tabs_create_mcp`). 기존 탭 재사용 금지(사용자가 명시 요청 시 예외).

### 2. 회사별 조회 (각 회사 반복)
1. 캐치 검색: `https://www.catch.co.kr/Comp/CompSearch?Word=<회사명또는도메인>` 으로 navigate.
2. 검색 결과에서 해당 기업 클릭 → 기업 상세(재무 탭) 진입.
3. 다음 데이터 수집:
   - 최근 3개년 **매출액** (추이 판단용)
   - 최근 **영업이익**(적자=마이너스 여부)
   - **직원수(인원)**
4. 페이지 텍스트는 `get_page_text` 또는 `read_page`로 추출해 수치를 읽는다.

### 3. 판정
- 위 "거름 기준" 적용 → ✅지원 / ❌거름 / ⚠️확인불가.
- 거름이면 **어느 기준에 걸렸는지** 사유를 명시.

### 4. 출력 (표 + 파일)
사용자에게 아래 표를 보여주고, 동일 내용을 `data/company-screening/<날짜>.md`에 저장한다.

| 회사 | 도메인 | 매출 추이(3개년) | 적자여부 | 인원 | 판정 | 사유 |
|------|--------|------------------|----------|------|------|------|
| 예시㈜ | example.com | 50→80→120억 (증가) | 적자 | 30명 | ✅지원 | 적자지만 매출 지속 증가 |
| 가나㈜ | gana.com | 30→28→25억 (감소) | 적자 | 12명 | ❌거름 | 적자+매출 감소 |
| 다라㈜ | dara.com | 100→110→120억 | 흑자 | 3명 | ❌거름 | 인원 5명 이하 |

판정 요약(지원 N건 / 거름 M건 / 확인불가 K건)을 표 아래 한 줄로 보고.

### 입력 경로 B 공통 출력

경로 B(잡사이트 저장 공고)는 위 표에 더해 다음을 적용한다:

- **마감일 열**을 추가한다(상시채용은 "상시", 이미 지난 공고는 "마감").
- **평균연봉 열**은 사이트가 제공할 때만 채운다(원티드=자체 제공, 사람인·잡코리아·점핏=미제공이라 보통 공란, 캐치에 있으면 캐치 값).
- **"마감 임박 공고" 섹션**을 별도로 마감일 오름차순 정리한다(상시·이미 마감 제외, 진행중 공고만). 오늘(currentDate) 기준 **남은 일수**를 표기하고, 당일/임박 건은 🔴 강조해 먼저 챙기도록 안내한다.
- 표 상단에 **데이터 출처** 한 줄을 명시한다(예: "출처: 사람인 스크랩(회사명·마감일) + 캐치(매출·영업이익·인원)").
- **파일명**:
  - 단일 사이트 — `data/company-screening/<날짜>-<사이트>-{bookmarks|scraps}.md` (원티드 `-wanted-bookmarks.md`, 사람인 `-saramin-scraps.md`, 잡코리아 `-jobkorea-scraps.md`, 점핏 `-jumpit-bookmarks.md`).
  - 여러 사이트 동시 — 통합 파일 `data/company-screening/<날짜>-saved-jobs.md` 하나에 표의 **출처(사이트) 열**을 추가하고, "마감 임박 공고" 섹션도 전 사이트 통합 오름차순으로 정리한다.

사이트별 수집·마감일 추출 상세는 각 reference 문서를 따른다.

## 에러 핸들링

- 캐치에서 기업을 못 찾으면(미등록/동명이인 등) → ⚠️확인불가, 사유에 "캐치 미등록" 표기. 다음 회사로 진행.
- 조회 2~3회 실패하거나 페이지 무응답이면 멈추고 사용자에게 상황 보고(무한 재시도 금지).
- 자바스크립트 alert/confirm 유발 요소 클릭 금지.
- 수치 단위(억/백만원) 혼동 주의 — 추이의 방향성이 핵심.

## 테스트 시나리오

**정상 흐름 (경로 A)**: 사용자가 도메인 5개 붙여넣기 → 캐치에서 각각 매출 3개년·영업이익·인원 조회 → 적자+매출감소 1건, 인원 5명 이하 1건 거름, 나머지 지원 판정 → 표 + `data/company-screening/2026-05-22.md` 저장 + "지원 3 / 거름 2" 요약.

**정상 흐름 (경로 B — 원티드)**: 사용자가 "크롬에서 원티드 북마크 찾아서 스크리닝해줘" → 크롬 연결·로그인 확인 → 원티드 저장 공고 목록에서 회사 20개 수집 → 각 회사 `__NEXT_DATA__`(인원·연봉) + `financial-report-for-wanted` API(매출·영업이익) 조회 → 동일 기준 판정 → 표(평균연봉 열 포함) + `data/company-screening/<날짜>-wanted-bookmarks.md` 저장 + 요약.

**정상 흐름 (경로 B — 사람인/잡코리아/점핏)**: 사용자가 "사람인 스크랩 걸러줘" → 로그인 확인 → 스크랩 페이지에서 회사명·마감일 수집(`references/saramin-scraps.md`) → 회사명 정규화 후 캐치로 매출·영업이익·인원 조회(경로 A 2단계) → 동일 기준 판정 → 표(마감일 열) + `data/company-screening/<날짜>-saramin-scraps.md` 저장 + "마감 임박 공고" 섹션 + 요약.

**정상 흐름 (경로 B — 여러 사이트)**: "내 저장 공고 다 정리해줘" → 로그인된 사이트(예: 사람인+점핏)를 차례로 수집 → 통합 캐치 조회 → 통합 파일 `data/company-screening/<날짜>-saved-jobs.md`(출처 열 포함) + 전 사이트 통합 마감 임박 섹션.

**에러 흐름 (경로 A)**: 도메인이 캐치 미등록 → ⚠️확인불가로 표기하고 사유 명시, 나머지 회사 정상 처리.

**에러 흐름 (경로 B)**: 크롬 미연결/해당 사이트 로그인 아님 → 자동 수집 불가를 알리고 경로 A로 폴백(회사명 직접 입력 요청). 특정 회사 캐치 미등록/API 빈 응답 → 해당 회사만 ⚠️확인불가, 나머지 계속. (잡코리아) 회사명이 잘려 복원 실패 → 해당 회사 ⚠️확인불가.
